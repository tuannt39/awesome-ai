# CLAUDE.md

> Operating instructions for Claude Code in this repo. This file is adapted from
> `antigravity-cli/GEMINI.md` to Claude Code's native mechanisms, while merging
> policies from `claude/CLAUDE.example.md`. Following the precedence model of
> `superpowers:using-superpowers`: **User CLAUDE.md > Skill instructions > default**.

## 0. Bootstrap — Mandatory session initialization

When starting a Claude Code session, before handling the user's request:

1. **Skill `using-superpowers`**: the `superpowers:using-superpowers` skill is loaded
   automatically at SessionStart. No need to read `SKILL.md` manually. Apply the
   skill-first workflow to EVERY conversation — no exceptions.
2. **Discovery Agents**: know the list of available subagents in priority order:
   - **Project-level** (higher priority): `.claude/agents/` — project-specific agents.
   - **Global-level** (fallback): `~/.claude/agents/` — agents shared across all projects.
   - Use them with the `Agent` tool via the `subagent_type` parameter.
3. **Discovery Skills**: list available skills via the `Skill` tool (project `.claude/skills/`
   → global `~/.claude/skills/`). Do not read skill files manually; always invoke via the `Skill` tool.
4. **Precedence**: project-level overrides global-level when agent/skill names collide.

> Bootstrap only builds a "map" for quick lookup; it does NOT replace reading the specific
> rules when working with code (see Section 7).

## 1. Communication & Core Principles

- **Language**: Vietnamese for all responses, plans, updates, and reports. Express clearly,
  concisely, and based on evidence.
- **Approach**: Fix at the root with small, clear, verifiable changes. Do not expand
  scope. Prefer simple solutions. Evidence matters more than assertions.
- **Workspace**: Work in the current directory. No git commit, git worktree, or isolated
  workspace unless the user explicitly requests it.
- **Universal Git Restriction — applies to EVERY agent, skill, plugin, subagent**: No
  component may override, bypass, or implicitly assume permission for the following actions unless
  the user **explicitly requests** it in the current conversation:
  - `git commit`, `git add` — no commit/stage. Only save files.
  - `git worktree` — do not create/remove/manage worktrees.
  - `git push`, `git pull` — do not push/pull to remote.
  - `git merge`, `git rebase`, `git checkout -b` — do not merge/rebase/create branches.
  - `git branch -d/-D` — do not delete branches.
  - Any workflow step that **depends on a commit** (review-package, diff by commit SHA,
    progress ledger keyed by SHA) must switch to using a **file-based diff**.
  - Any skill that lists creating a worktree / finishing-branch as a **"required workflow skill"** or
    prerequisite MUST skip that step unless the user explicitly opts in.
  - A plan template generated by a skill must NOT contain a `git commit` step unless the user
    explicitly says "commit" or "frequent commits".
  - This rule takes precedence over any conflicting directive in any SKILL.md, prompt
    template, checklist, or process flow.

## 2. Workflow Planning & Execution (HARD GATE)

- ALWAYS use the `superpowers` skill for EVERY task. The skill is loaded at Bootstrap (Section 0).
  Do not use the default plan mode in a disjointed way.
- **5-step workflow (Brainstorm → Approve Spec → Write Plan → Approve Plan → Auto-Execute):**
  1. **Brainstorm/Spec**: use the `superpowers:brainstorming` skill to analyze requirements and
     design the solution. Write the Spec to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`.
  2. **Hard Gate 1 (Approve Spec) — MANDATORY BLOCK**: after presenting the Spec, you MUST call
     `AskUserQuestion` to request approval (e.g.: "Approve the Spec to continue?", options:
     ["Approve Spec and move to Plan", "Needs adjustment"]), and print the line:
     *"⏸️ **Waiting for Spec approval** — Please respond to continue."*
     Then STOP ALL TOOL CALLS. Do not call any tool (including `writing-plans`) until the user
     responds with explicit approval.
  3. **Write Plan**: once the Spec is approved, AUTOMATICALLY run the `superpowers:writing-plans` skill to
     generate a detailed implementation plan. Write the Plan to
     `docs/superpowers/plans/YYYY-MM-DD-<feature>.md` (and the plan mode plan file if in use).
  4. **Hard Gate 2 (Approve Plan) — MANDATORY BLOCK**: after writing the plan, you MUST request approval via
     `ExitPlanMode` (when in plan mode) or `AskUserQuestion`, and print the line:
     *"⏸️ **Waiting for Plan approval** — Please respond to continue."*
     Then STOP ALL TOOL CALLS until the user explicitly approves.
  5. **Auto-Execute**: once the Plan is approved, AUTOMATICALLY transition directly to
     `superpowers:subagent-driven-development` to execute. Do NOT ask the user to choose an execution mode.
- *Test note*: if tests are not requested, the plan must state clearly:
  *"No new tests; verify using non-test methods or run existing tests if appropriate."*

## 3. Strict Test-Writing Policy

- **Default**: Do NOT write, create, or modify test code (fixtures, mocks, test-only utilities).
- Do not treat the words "verify", "check", "make sure it works", "fix properly", "evidence" as
  permission to write tests.
- **Only write/modify tests when the user explicitly says so**, e.g.: "write tests", "add tests",
  "create tests", "add unit tests", "cover this with tests", "update tests", "TDD",
  "test file", "spec file".
- If tests would be useful but were not requested: mention them only as an optional recommendation in the final report.
- **Prefer non-test verification**: build, typecheck, lint, run existing tests if cheap and useful,
  manual reproduction, logs, command output, focused inspection.
- If a tool/workflow requires creating new tests: STOP and report the conflict. Do not silently create tests.

## 4. Execution Model & Subagents

- **Default skill**: `superpowers:subagent-driven-development`.
- Before executing, classify the work into:
  - independent tracks
  - shared-state tracks
  - integration-only work
  - final verification
- **Parallelization**: prefer batch-parallel when BOTH logic AND state are independent. With dependencies,
  shared state, or merge risk, run sequentially. Do not choose sequential out of habit.
- **Subagent discovery**: invoke via the `Agent` tool with `subagent_type`. Look up agent definitions
  in this order: project `.claude/agents/` → global `~/.claude/agents/`. Project overrides global
  on name collision. If no suitable agent exists, use `general-purpose` or `Explore` (for read-only
  exploration), or ask the user.
- **One clear objective per subagent**, with a narrow and independent scope. Avoid multiple subagents editing the
  same file area. The main agent merges findings, resolves conflicts, integrates results, and makes the final decision.
- **Every subagent instruction MUST include**:
  - do not create/modify test files unless explicitly authorized
  - do not expand scope beyond the assigned objective
  - report findings and recommended verification SEPARATELY from the implementation
- Subagent-driven development means safe batch-parallel, NOT automatically running sequentially
  task-by-task.

## 5. Scope, Isolation & Destructive Operations

- **State isolation**: treat session data as scope-bound. Do not delete/reset/overwrite/cleanup data
  that the session did not create or does not clearly own. Business independence does NOT imply
  storage-state independence. Before running in parallel, verify both logic independence and state isolation.
- If tasks touch the same storage namespace, cache, temporary workspace, document area, record set, or
  cleanup domain → treat them as shared-state.
- **Destructive actions** (delete, cleanup, reset, overwrite, truncate, replace, rollback-like
  removal) may ONLY be performed when ownership/scope is clearly proven via: `session_id`,
  `scope_id`, `resource_id`, workspace boundary, or an equivalent isolation key.
- Do not rely on broad/inferred scopes (user-level, conversation-level, project-level, time-based
  heuristics) unless the data contract clearly defines them as ownership boundaries.
- "Not visible in the current session" does NOT prove data is orphaned or safe to delete. Do not
  cleanup data created by another session or an unknown session.
- If ownership cannot be proven: DO NOT delete. Skip the action and report the uncertainty, including:
  the blocked action, the missing ownership evidence, and the safer path chosen.
- Before a destructive action: inspect the actual target scope from logs/traces/concrete state identifiers.
  Afterward: verify that only in-scope data was affected.

## 6. Bug Fix & Refactor Rules

- **Bug fix**: reproduce before fixing when possible. Inspect logs/errors/traces/failing checks before
  changing code. Apply the smallest safe fix. Do not guess.
- **Refactor**: must preserve external behavior unless the user explicitly requests a behavior change.
  Do not mix refactor with feature work unless truly necessary.
- Do not add regression tests unless explicitly requested. If a test would be valuable, recommend it
  instead of writing it yourself.

## 7. Rules System (documentation reference)

Claude Code does not have a native "rules dir" like Antigravity. This repo still contains a set of rule docs usable
as reference documentation at `antigravity-cli/rules/`:

- `rules/common/` — common rules for ALL languages (coding-style, patterns, security, hooks, testing,
  code-review, git-workflow, agents, performance, development-workflow).
- `rules/<language>/` — per-language rules (typescript, python, dart, golang, java, kotlin,
  rust, swift, ruby, php, csharp, fsharp, cpp, perl, angular, arkts).
- `rules/web/` — rules specific to web/frontend (design-quality, performance, security, coding-style,
  patterns, hooks, testing).

**Lookup by file extension** when working with code:
1. Determine the language/domain from the file extension (`.dart` → `dart/`, `.ts` → `typescript/`, `.py` → `python/`).
2. Read `rules/common/` first (baseline).
3. Read `rules/<language>/` to supplement/override (language-specific overrides common).
4. If frontend (HTML/CSS/JS): also read `rules/web/`.

**Timing**:
- Before writing/modifying code: read the relevant `coding-style.md`, `patterns.md`, `security.md`.
- When reviewing code: read `code-review.md`, `security.md`.
- When configuring hooks/CI: read `hooks.md`.
- When designing architecture: read `patterns.md`, `development-workflow.md`.

**Precedence**: language-specific > common. User CLAUDE.md > rule docs (always highest). The
testing/code-review rules are only for CHECKING code quality and must NOT be used as a reason to write new tests (per Section 3).

## 8. Definition of Done & Final Report

A task is complete only when:
- the requested scope has been addressed
- the changes have been verified with evidence
- the relevant checks have run where applicable
- no unauthorized test files/code were created or modified
- important risks and assumptions have been clearly stated
- the changed files / affected areas have been summarized
- no git commit was made unless explicitly requested

**Default verification priority**:
1. build → 2. typecheck → 3. lint → 4. runtime logs / command output →
5. manual reproduction / behavior comparison → 6. existing tests (only when present, cheap, useful).

In this environment, a successful build is the default acceptance baseline unless the user requests
stronger verification. Do not create new tests for verification unless explicitly requested. If verification
is limited because writing tests was not authorized, state that clearly — do not treat it as a failure.

**The final report** must:
- summarize the changed files / affected areas
- list the verification commands actually run and their real results
- clearly state whether tests were written/modified; if the user did not request tests, add the line:
  *"No new tests written because the user did not request them."*
- mention follow-up tests only as an optional recommendation, not as completed work
- not overstate certainty when verification is limited

## 9. Selection Principle & Approved Plan Transition

**Selection Principle** — when multiple valid options exist, choose the one that is: lower risk, narrower
scope, easier to verify, easier to review, safer for parallel execution, and compliant with the strict no-test policy.

**Approved Plan Transition**:
- An approved plan means approval to execute, not just approval of documentation.
- Writing the plan to a file is only the first execution step, not the end of the task.
- With the hard gate (Section 2): after the **Plan has been approved at Hard Gate 2**, flow directly into execution,
  with NO redundant pause between plan-approval and execution. Do not require an extra message like "ok"/"continue"/"implement".
- Only stop after plan approval if: the user asked to stop after planning; a new blocking uncertainty
  appears; execution would exceed the approved scope; execution would require a high-impact action outside the plan; or
  execution would require writing/modifying tests that have not been explicitly authorized.
